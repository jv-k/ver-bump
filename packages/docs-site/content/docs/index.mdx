**VerBump is a plain-bash release tool for any Git repo.** It reads your
[Conventional Commits](https://www.conventionalcommits.org/) to suggest the next
[SemVer](https://semver.org/), then updates the changelog, bumps your version
files, tags, and pushes — with `git` and `jq` as the only runtime dependencies.

## Why VerBump?

Cutting a release shouldn't require installing a bigger toolchain than the thing
being released.

Release tooling has drifted into two camps: fully automated CI machinery like
[semantic-release](https://github.com/semantic-release/semantic-release) —
powerful, but Node-only, deliberately prompt-free, and a deep dependency tree
for what is ultimately a git tag — and single-purpose bumpers like
[bump-my-version](https://github.com/callowayproject/bump-my-version) that
rewrite a version string and stop. VerBump is the middle of that spectrum, for
every repo and not just the Node ones: a tool that reads your Conventional
Commits and *suggests* the right SemVer bump, then writes the changelog, tags,
pushes, and opens the PR or GitHub release — with every side-effect previewable
via `--dry-run`, reversible via `--undo`, and nothing to install beyond
standard CLI tools.

## Features

| Feature | Description |
| --- | --- |
| <Feature>Zero real dependencies</Feature> | Pure bash. Only `git` and `jq` are needed to run it. |
| <Feature>Multi-format file bumps</Feature> | Keeps `package.json`, `pyproject.toml`, a Go const, a Helm chart, or any text file in sync with the tag via `--bump`. |
| <Feature>Smart bump suggestion</Feature> | Reads Conventional Commits since the last tag to propose **major** / **minor** / **patch**, and advances prerelease counters (`4.0.0-dev.6` → `4.0.0-dev.7`). |
| <Feature>Automatic CHANGELOG</Feature> | Generates and updates `CHANGELOG.md`: a flat list, or Conventional-Commit-**grouped** sections with commit/PR/compare links. |
| <Feature>Three release workflows</Feature> | Tag-in-place (default), release branch (`--branch`), or release PR (`--pr`). Pick per-run or set a default. |
| <Feature>Monorepo packages</Feature> | Per-package tag series and path-scoped commit analysis: run VerBump from a package dir and the suggestion, changelog, and release notes cover only that package. |
| <Feature>Safety preflights</Feature> | Refuses to release on a dirty tree, an out-of-sync remote, or a disallowed branch — each individually overridable. |
| <Feature>Dry-run</Feature> | `--dry-run` prints every side-effect (file write, `git add`, commit, tag, push) without executing any of them. |
| <Feature>Undo</Feature> | `--undo` rolls back a local release (tag + release branch) before anything is pushed. |
| <Feature>GitHub releases & PRs</Feature> | `--release` publishes a GitHub release for the new tag; `--pr` opens a pull request. Both use the optional [`gh`](https://cli.github.com) CLI. |
| <Feature>Release hooks</Feature> | `PRE_BUMP_CMD` / `POST_TAG_CMD` run your tests before the bump and build artifacts after the tag. |
| <Feature>Signed tags</Feature> | Annotated tags by default; `--sign` produces GPG-signed tags using your git config. |
| <Feature>Shell completions</Feature> | Built-in completion scripts for **bash**, **zsh**, and **fish**. |
| <Feature>SemVer 2.0 validation</Feature> | Every version input is validated against the SemVer 2.0 spec, including `-prerelease` and `+build` metadata. Typos fail fast. |

### How it compares

| Capability | VerBump | [release-it](https://github.com/release-it/release-it) | [semantic-release](https://github.com/semantic-release/semantic-release) | [release-please](https://github.com/googleapis/release-please) | [changesets](https://github.com/changesets/changesets) |
| --- | --- | --- | --- | --- | --- |
| Runtime dependencies | **bash + git + jq** | Node | Node + plugin stack | Node / GitHub App | Node |
| Interactive with a smart suggestion | <Yes /> | <Yes /> | <No /> CI-only | <No /> PR-driven | <Partial /> intent files |
| Dry-run preview of every side-effect | <Yes /> `--dry-run` (+ `--json`) | <Yes /> | <Partial /> | n/a | <Partial /> |
| Local undo before pushing | <Yes /> `--undo` | <No /> | <No /> | n/a | <No /> |
| Monorepo: per-package tags | <Yes /> [`TAG_PREFIX`](/docs/guides/monorepos) | <Yes /> | <Partial /> plugins | <Yes /> | <Yes /> |
| Monorepo: scoped suggestion + changelog | <Yes /> [`COMMIT_PATHS`](/docs/guides/monorepos) | <Partial /> partial | <Partial /> plugins | <Yes /> | <Yes /> |
| Monorepo: scoped release notes | <Yes /> | <No /> | <Partial /> | <Yes /> | <Yes /> |
| Monorepo-aware dirty-tree check | <Yes /> | <No /> | n/a | n/a | <No /> |
| Workspace dependency-graph bumping | <No /> by design | <No /> | <Partial /> plugins | <Partial /> partial | <Yes /> |
| "Release everything that changed" | <No /> [your scripts](/docs/guides/monorepos) | <No /> | <Yes /> per-package CI | <Yes /> release PR | <Yes /> |

VerBump aims to be the strongest *manual, per-package* release tool with zero
Node runtime. If you want fully automated change detection or interdependent
workspace versioning, the last two columns are the right tools — and we say
so in the [monorepos guide](/docs/guides/monorepos).

## How it works

A single `verbump` run walks through five phases:

| Phase | What happens |
| --- | --- |
| 1. **Verify** | Confirms commits exist, the working tree is clean, the remote is in sync, and the current branch is allowed to release. |
| 2. **Choose a version** | Suggests the next SemVer from your Conventional Commits, or takes an explicit `-v <version>`, a forced `--major` / `--minor` / `--patch`, or a prerelease `--preid <id>`. |
| 3. **Bump** | Writes the new version into `package.json` (and any `--bump` targets), then regenerates `CHANGELOG.md`. |
| 4. **Commit & tag** | Commits the changes on the current branch and creates an annotated (or `--sign`ed) tag. |
| 5. **Push & publish** | Optionally pushes the commit and tag. With `--pr` / `--release` it opens a pull request or a GitHub release. |

Every side-effecting step honours `--dry-run`, and preconditions fail with a
[documented exit code](/docs/reference/exit-codes) and an actionable hint.
